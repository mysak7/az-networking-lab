Peering Topology

  1. 1 Hub ↔ App Spoke and Hub ↔ Mgmt Spoke are each a bidirectional peering pair (allow_forwarded_traffic = true).
  2. 2 There is no direct peering between App Spoke and Mgmt Spoke. Traffic between spokes must traverse the hub.
  3. 3 A future Azure Firewall or NVA in sn-shared can inspect and filter all cross-spoke flows via UDRs.
  4. 4 New spokes are added by peering to the hub only — no mesh updates needed across existing spokes.

Design Rationale & What This Lab Provides

Pattern
Hub-and-spoke — the standard Azure enterprise network topology
No VMs or actual workloads
This is a pure networking foundation. VNets, peerings, NSGs, and DNS zones are provisioned and ready; no compute is deployed.
Address space sizing
/16 per VNet gives 65 536 IPs per VNet — deliberate over-provisioning so future subnet slicing does not require re-addressing
Region
East US 2 (all resources in a single region for cost and latency)
Extensibility
Add spoke VNets by peering to the hub; DNS zones are already linked so private endpoints work immediately
Cost profile
Near-zero — VNet peering charges apply only when data traverses the peering. No VMs, no gateways, no firewalls.
Next modules can add
Azure Firewall (sn-shared), VPN/ER Gateway, Bastion (sn-shared), Private Resolver (sn-dns), AKS in sn-app, Storage/KeyVault with PEPs in sn-pep
IAC
Terraform — single main.tf, variables.tf, outputs.tf; no modules needed at this scale
