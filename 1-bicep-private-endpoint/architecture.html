How Private DNS Resolution Works

  1. 1 The VM runs nslookup milearnnetdiag.blob.core.windows.net. The query goes to Azure's internal DNS resolver (168.63.129.16).
  2. 2 Azure DNS checks the VNet for linked Private DNS Zones. It finds privatelink.blob.core.windows.net linked via mi-dns-link.
  3. 3 The storage FQDN is a CNAME to milearnnetdiag.privatelink.blob.core.windows.net. The DNS Zone Group has registered an A-record in the private zone pointing to 10.0.2.x.
  4. 4 DNS responds with the private IP 10.0.2.x — not the public Azure storage endpoint. The VM connects to storage over the Azure backbone, never leaving the VNet.
  5. 5 Traffic flows: VM (10.0.1.x) → Private Endpoint NIC (10.0.2.x) → Storage Account. The storage account's public-access deny rule is irrelevant — the connection arrives from a private IP.

Bicep vs Terraform — Key Differences

State management
Bicep: no state file — ARM manages desired state. Terraform: local/remote .tfstate required.
NSG-to-NIC association
Bicep: inline inside the NIC resource (networkSecurityGroup.id). Terraform: separate azurerm_network_interface_security_group_association resource.
Subnet definition
Bicep: child resources of the VNet (parent: vnet). Terraform: separate top-level resources.
AAD / Service Principal
Bicep: not supported natively — use setup-sp.sh script separately. Terraform: first-class resources via azuread provider.
Deployment scope
Bicep: targetScope = 'subscription' creates the RG inline. Terraform: provider-level subscription config.
Outputs
Bicep: output declarations, accessed via az deployment sub show. Terraform: terraform output.

  sshPublicKey="$(cat ~/.ssh/id_rsa.pub)"

  --location eastus2 \
  --template-file main.bicep \
  --parameters main.bicepparam \

az deployment sub show \
  --name main \
  --query properties.outputs
