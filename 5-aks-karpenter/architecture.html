Karpenter / NAP Scaling Flow

  1. 1 Pending pod detected — scheduler cannot place pod; no node has capacity or matches constraints
  2. 2 Karpenter evaluates NodePool CRD — checks requirements (instance type, zone, arch, capacity type)
  3. 3 AKSNodeClass resolved — determines OS image, disk size, subnet, and labels for new node
  4. 4 Node provisioned — Azure VM starts, joins cluster via kubelet, node becomes Ready
  5. 5 Pod scheduled & running — scheduler places pod on the new node; workload is live
  6. 6 Idle detection — Karpenter monitors node utilisation; empty or underutilised node identified
  7. 7 Consolidation / removal — pods drained, node cordoned, VM deleted; cluster right-sized automatically

Azure CNI Overlay

What it solves
Classic Azure CNI assigns one VNet IP per pod — exhausts subnets at scale. Overlay decouples pod IPs from VNet.
How it works
Pods get IPs from a private pod CIDR (192.168.0.0/16). Traffic is encapsulated at the host and routed through the node's VNet NIC.
Node IPs
Only nodes consume VNet IP space (10.10.1.x). Pods are invisible to the VNet.
Scale headroom
65 536 pod IPs available without changing VNet addressing.

NAP vs. Cluster Autoscaler

Cluster Autoscaler
Scales pre-defined node pools up/down. You must define instance types and pool sizes in advance.
NAP / Karpenter
Just-in-time node provisioning. Karpenter picks the optimal instance type per workload request automatically.
NodePool CRD
Replaces static node pool definitions. Declare constraints; Karpenter decides the VM.
Cost efficiency
Bin-packing + spot support + rapid scale-down = lower cloud bill vs. Cluster Autoscaler.

Workload Identity

Problem solved
Pods need Azure credentials (Key Vault, Storage, etc.) without storing secrets in-cluster.
OIDC Issuer
AKS exposes an OIDC endpoint. Azure AD trusts tokens signed by this issuer.
Federation
A Kubernetes ServiceAccount annotated with a Managed Identity client ID can exchange its token for an Azure AD token.
Result
Pods call Azure APIs with zero secrets — identity flows from pod SA → Azure AD → resource.
